# Supabase Keep-Alive Workflow

[![GitHub Actions Status](https://img.shields.io/github/actions/workflow/status/your-username/your-repo/keep-alive.yml?branch=main)](https://github.com/your-username/your-repo/actions)

A simple, robust GitHub Action workflow that prevents your Supabase project from pausing due to inactivity. It keeps your project awake even if there is no real usage, by periodically inserting a dummy record into your database.

## 🌟 How It Works

Supabase automatically pauses free-tier projects if there is no activity for a week. This repository solves that problem by running a GitHub Actions workflow every 2 days.

**The Workflow Process:**
1. Triggers automatically based on a cron schedule (`0 0 */2 * *`), meaning it runs every 2 days at midnight. You can also trigger it manually.
2. Sends an HTTP POST request using the Supabase REST API.
3. Inserts an empty JSON payload `{}` into a dedicated `keep_alive_logs` table. The database automatically generates the `id` and `run_time` timestamp.
4. **Output:** Evaluates the HTTP response. It outputs a success message if the insert succeeds (HTTP 200 or 201), or fails the workflow and triggers GitHub's default email notification if an error occurs.

## 🛠️ Prerequisites & Setup

To use this workflow in your project, follow these simple steps:

### 1. Create the Database Table
Run the following SQL command in your Supabase SQL Editor to create the required table:

```sql
create table public.keep_alive_logs (
  id bigint generated by default as identity primary key,
  run_time timestamp with time zone default now()
);
```

### 2. Configure GitHub Secrets
Go to your GitHub repository settings: **Settings > Secrets and variables > Actions > New repository secret**.

Add the following two secrets:
- `SUPABASE_URL`: Your Supabase Project URL (e.g., `https://xxxx.supabase.co`). You can find this in your Supabase project settings under **API**.
- `SUPABASE_SERVICE_ROLE_KEY`: Your Supabase Service Role Key. Found in the same **API** settings page.

### 3. Add the Workflow
Copy the `.github/workflows/keep-alive.yml` file from this repository to the `.github/workflows/` directory in your own repository.

## 🔐 Security Considerations: Service Role Key vs. Anon Key

By default, this workflow is configured to use the `SUPABASE_SERVICE_ROLE_KEY`.

**Why use the Service Role Key?**
We use the Service Role Key to guarantee that the script works out-of-the-box. The Service Role Key bypasses Row Level Security (RLS) policies in Supabase, meaning the insert operation will definitely succeed without requiring you to configure complex security policies for this simple table.

**Alternative Approach (More Secure)**
If you are strictly following the principle of least privilege and prefer not to expose your Service Role Key in GitHub Actions, you can use the `SUPABASE_ANON_KEY` instead.

To do this:
1. Replace `SUPABASE_SERVICE_ROLE_KEY` with `SUPABASE_ANON_KEY` in your GitHub Secrets.
2. Update the workflow file `.github/workflows/keep-alive.yml` to use `SUPABASE_ANON_KEY`.
3. Enable RLS and add a policy to allow inserts from anonymous users:

```sql
-- Enable RLS on the table
alter table public.keep_alive_logs enable row level security;

-- Create a policy allowing anonymous inserts
create policy "Allow anonymous inserts for keep-alive"
  on public.keep_alive_logs
  for insert
  to anon
  with check (true);
```
