# إجراء الإبقاء على نشاط Supabase

[![GitHub Actions Status](https://img.shields.io/github/actions/workflow/status/your-username/your-repo/keep-alive.yml?branch=main)](https://github.com/your-username/your-repo/actions)

إجراء (GitHub Action) بسيط وفعّال يمنع إيقاف مشروعك في Supabase بسبب الخمول. يقوم هذا الإجراء بجعل مشروعك يبقى مستيقظاً حتى وإن لم يكن هناك أي استخدام حقيقي للمشروع، وذلك عن طريق إدراج سجل وهمي في قاعدة البيانات بشكل دوري.

## 🌟 كيف يعمل المشروع؟

تقوم منصة Supabase بإيقاف المشاريع المجانية مؤقتاً إذا لم يكن هناك أي نشاط لمدة أسبوع. هذا المستودع يحل هذه المشكلة من خلال تشغيل إجراء (Workflow) عبر GitHub Actions كل يومين.

**خطوات سير العمل:**
1. يتم تشغيل الإجراء تلقائياً بناءً على جدول زمني (`0 0 */2 * *`)، مما يعني أنه يعمل كل يومين عند منتصف الليل. يمكنك أيضاً تشغيله يدوياً.
2. يرسل طلب HTTP POST باستخدام واجهة برمجة تطبيقات Supabase REST API.
3. يقوم بإدراج بيانات JSON فارغة `{}` في جدول مخصص يُسمى `keep_alive_logs`. تقوم قاعدة البيانات بتوليد المُعرف (`id`) ووقت التشغيل (`run_time`) تلقائياً.
4. **المخرجات:** يتم تقييم استجابة الـ HTTP. يخرج رسالة نجاح إذا تم الإدراج بنجاح (حالة HTTP 200 أو 201)، أو يفشل الإجراء ويُشغل إشعارات البريد الإلكتروني الافتراضية من GitHub في حال حدوث خطأ.

## 🛠️ المتطلبات والإعدادات

لاستخدام هذا الإجراء في مشروعك، اتبع هذه الخطوات البسيطة:

### 1. إنشاء جدول في قاعدة البيانات
قم بتشغيل أمر SQL التالي في محرر SQL الخاص بـ Supabase لإنشاء الجدول المطلوب:

```sql
create table public.keep_alive_logs (
  id bigint generated by default as identity primary key,
  run_time timestamp with time zone default now()
);
```

### 2. إعداد أسرار GitHub (GitHub Secrets)
اذهب إلى إعدادات مستودعك في GitHub: **Settings > Secrets and variables > Actions > New repository secret**.

قم بإضافة السرّين التاليين:
- `SUPABASE_URL`: رابط مشروع Supabase الخاص بك (مثل `https://xxxx.supabase.co`). يمكنك العثور عليه في إعدادات مشروعك تحت قسم **API**.
- `SUPABASE_SERVICE_ROLE_KEY`: مفتاح الـ Service Role لمشروعك. يوجد في نفس صفحة إعدادات **API**.

### 3. إضافة ملف الإجراء (Workflow)
انسخ ملف `.github/workflows/keep-alive.yml` من هذا المستودع وضعه في مجلد `.github/workflows/` في مستودعك الخاص.

## 🔐 اعتبارات أمنية: مفتاح Service Role مقابل مفتاح Anon

بشكل افتراضي، تم إعداد هذا الإجراء لاستخدام `SUPABASE_SERVICE_ROLE_KEY`.

**لماذا نستخدم مفتاح Service Role؟**
لقد استخدمنا مفتاح Service Role كتأكيد على أن السكربت سيعمل بشكل أكيد وبدون مشاكل. هذا المفتاح يتجاوز قواعد أمان مستوى الصفوف (RLS) في Supabase، مما يعني أن عملية الإدراج ستنجح حتماً دون الحاجة إلى إعداد قواعد أمان معقدة لهذا الجدول البسيط.

**الطريقة البديلة (أكثر أماناً)**
إذا كنت تتبع مبدأ "الامتياز الأقل" بشكل صارم وتفضل عدم تعريض مفتاح Service Role الخاص بك في GitHub Actions، يمكنك استخدام `SUPABASE_ANON_KEY` بدلاً من ذلك.

للقيام بذلك:
1. استبدل `SUPABASE_SERVICE_ROLE_KEY` بـ `SUPABASE_ANON_KEY` في إعدادات أسرار GitHub.
2. قم بتحديث ملف الإجراء `.github/workflows/keep-alive.yml` لاستخدام `SUPABASE_ANON_KEY`.
3. قم بتفعيل قواعد RLS وأضف سياسة تسمح بعمليات الإدراج للمستخدمين المجهولين (anonymous):

```sql
-- تفعيل قواعد أمان مستوى الصفوف (RLS) على الجدول
alter table public.keep_alive_logs enable row level security;

-- إنشاء سياسة تسمح بعمليات الإدراج للمستخدمين المجهولين
create policy "Allow anonymous inserts for keep-alive"
  on public.keep_alive_logs
  for insert
  to anon
  with check (true);
```
